# ADR-007: Explicit First-Seen Conversation Message Ordering

**Status**: Accepted
**Date**: 2026-04-06
**Ticket**: `specs/tickets/017-canonical-conversation-message-ordering/`

## Context

Ticket 016 established a core-owned conversation projector and Murmur-owned update contract, but it did not fully settle how canonical visible message order should be defined.

Today Murmur still mixes several different ordering proxies:

- `WorkspaceState.unified_timeline/2` sorts merged messages by `message.id`
- `WorkspaceLive` still uses local insertion and replacement heuristics to keep busy-session messages visually stable
- assistant display ids are often synthesized from `request_id <> "-turn"`
- user display ids may come from persisted thread-entry ids or local UUIDv7 fallbacks

Those proxies are not equivalent to "the time this visible message was first created."

They are also not backed by one reliable source of temporal identity:

- persisted thread entries are stored in `jido_murmur_thread_entries` with an autogenerated Ecto `:binary_id`, so the persisted entry id is not a canonical time-ordered UUIDv7 created upstream
- `request_id` is a Murmur-generated UUIDv7 for a run, but one ReAct run may contain multiple internal assistant/tool phases
- projected display ids may be preserved from a thread entry, synthesized from a request id, or generated locally when no id is present

Implementation work also exposed a second mismatch: Murmur currently collapses an entire outer ReAct request into one assistant message in both the live read model and the persisted-entry projection path. That is too coarse for the desired steering behavior.

The intended product behavior is:

- once an LLM invocation starts, later steering messages cannot influence that already-running assistant/tool phase
- those steering messages can influence the next assistant phase within the same outer request
- therefore a steering message sent during a run should appear after the current assistant phase and its tool outcomes, but before the next assistant phase

Fully ordering every visible sub-element would model this precisely, but it would also add unnecessary complexity. The simpler and cleaner rule is to keep ordering at the top-level message layer while changing what an assistant message represents.

## Decision

Murmur should define canonical conversation ordering by explicit first-seen creation order over top-level conversation messages rather than by `request_id`, display id, or LiveView-local insertion heuristics.

Specifically:

- every canonical top-level conversation message must carry or derive one explicit first-seen ordering key owned by Murmur's conversation projector and read model
- that ordering key represents when the top-level message first entered canonical conversation state, not when a later update modified its content
- human messages and inter-agent `tell` messages remain ordinary top-level user messages
- assistant messages are segmented by assistant step, where one assistant step means one LLM invocation plus the tool calls and tool results produced before the next LLM invocation or request completion
- one outer `request_id` may therefore produce multiple assistant messages over time
- tool calls and tool results remain nested within the assistant step that produced them rather than becoming separately ordered top-level items
- snapshot loading and incremental updates must preserve the same top-level assistant-step ordering semantics
- the implementation must stay read-model scoped within Murmur's conversation projector, read model, and persisted-entry projection boundary
- Murmur must not require `jido_ai` strategy, directive, or runtime contract changes to implement canonical ordering
- UIs should render the canonical order provided by core and should stop inferring position from current rendered state
- `request_id` remains outer run identity, not a universal sort key or the identity of one canonical assistant message
- Murmur owns the identity of assistant-step messages and any deterministic secondary tie-breaker needed for stable ordering
- Murmur should not preserve a transitional compatibility layer for legacy ordering code once the canonical rule is adopted; if breaking backwards compatibility yields a cleaner end state, the cleaner end state is preferred

When two top-level messages share the same coarse timestamp, Murmur should apply a deterministic secondary tie-breaker owned by core so ordering remains stable across reconnects and refreshes.

## Consequences

Benefits:

- split and unified views can follow one ordering rule without local heuristics
- the product rule becomes explicit: order top-level messages by first creation time, not by id shape or UI insertion state
- the execution model and the visible model align more closely without forcing a fully ordered list of sub-elements
- reconnect and refresh behavior become easier to reason about because the same canonical ordering metadata survives projection boundaries
- future host UIs can render canonical assistant-step state without re-implementing insertion logic
- the agent loop and `jido_ai` strategy boundary stay unchanged because ordering is owned on Murmur's read side
- required `UITurn` retirement work can land in the same pass rather than through a staged compatibility layer

Trade-offs:

- Murmur must add and maintain explicit ordering metadata plus assistant-step projection state in its read model rather than relying on existing ids
- live and persisted projection code will both need to stop collapsing entire requests into one assistant message
- some current helper and rendering code will need to be rewritten to consume core-owned ordering instead of sorting by id or preserving append position
- tests will need to cover assistant-step segmentation separately from stable identity and content updates
- downstream code that currently depends on legacy ordering behavior may need to change in one step rather than through a compatibility phase
