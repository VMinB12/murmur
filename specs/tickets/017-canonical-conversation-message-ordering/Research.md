# Research: Canonical Conversation Step Ordering

## Objective

Determine whether Murmur can safely order visible conversation messages by existing ids, and if not, what canonical ordering rule should replace the current mix of id sorting and view-local heuristics.

## Findings

### Persisted thread-entry ids are not canonical time-ordered message ids

`JidoMurmur.Storage.ThreadEntry` uses an autogenerated Ecto `:binary_id` primary key, and `JidoMurmur.Storage.Ecto.append/3` inserts thread rows without preserving any upstream entry id. When rows are reconstructed into thread entries, the returned entry id is the database row id.

This means persisted entry ids are not a reliable chronological ordering key for visible messages.

### Display message ids mix several sources

`UITurn` builds user display messages from `entry.id` when available and falls back to local UUIDv7 generation when not. Assistant display messages use `request_id <> "-turn"` when a request id exists, and otherwise derive from the first entry id.

As a result, visible `DisplayMessage.id` values do not all follow one provenance or one temporal format.

### `request_id` is a stable turn id, not a per-message ordering key

`JidoMurmur.Runner` generates `request_id` with UUIDv7, which makes it suitable for identifying one assistant run. It does not identify every visible message that can appear while that run is active, including human messages that arrive during the run.

### Current ordering is still partly UI-owned

`WorkspaceState.unified_timeline/2` sorts by `message.id`, while `WorkspaceLive` uses `insert_user_message/3`, `find_last_running_assistant_index/1`, and `upsert_message/2` to preserve a user-visible order during active runs.

These rules are deterministic, but they are still proxies rather than an explicit canonical statement of "first created message order."

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| Sort by `request_id` | Reuses stable assistant-turn identity and is simple for one run | Not defined for every visible message and fails when a human message appears during an active run |
| Sort by `DisplayMessage.id` or thread-entry id | Minimal code churn and deterministic | Ids come from mixed sources and persisted entry ids are autogenerated database ids rather than canonical time-ordered message ids |
| Add explicit first-seen ordering metadata to canonical messages | Matches product intent, removes UI heuristics, and survives snapshot/update boundaries | Requires read-model changes, tests, and migration of current sorting code |

## Recommendation

Adopt explicit first-seen ordering metadata at the core conversation projector boundary.

The canonical conversation model should own a message ordering key that reflects when the visible message first entered canonical state, with a deterministic tie-breaker for equal timestamps. UI consumers should render that order directly instead of deriving it from ids or insertion heuristics.
